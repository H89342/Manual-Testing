Part B: New Feature Implementation
The team’s discussion raised five planned features with open design questions. Below is detailed technical guidance for each, including endpoint designs, response schemas, database considerations, and Claude Code workflows.
1. Engine Codes & DTC Codes
Open questions: Should this be a new endpoint? Are engine codes the same as DTC codes?
Answer: Engine codes and DTC (Diagnostic Trouble Code) codes are the same standard — both follow SAE J2012/ISO 15031-6. The existing endpoint GET /api/devices/{deviceId}/dtc-codes already returns these but, critically, the plain-English reference confirms that “the description and severity fields are typically not populated.”
Key Finding: The Front-End Already Has This Data
The Telemax live map UI already displays rich DTC information in its notification panel. For example, when a P1094 fault is detected, the front-end shows:
“Low voltage signal from the Bank 1, Sensor 2 oxygen sensor (located after the catalytic converter)”Description — 
An explanation of the sensor’s role (monitors exhaust gases for emissions compliance) and the consequences of not addressing the issue (reduced fuel efficiency, engine/emission system damage).Why this matters — 
A list of likely root causes: faulty oxygen sensor, damaged or corroded wiring/connections, exhaust leak near sensor, catalytic converter issues.Possible causes — 
Numbered steps: 1) Schedule vehicle inspection, 2) Check for other related DTC codes, 3) Inspect oxygen sensor wiring/connectors, 4) Repair or replace faulty parts.Recommended action — 
A “Critical” tag displayed prominently on the notification.Severity badge — 
Where and when the fault was detected (e.g., “11 Gould Rd, Herston QLD AUS, 18/02/2026 12:15:46 PM”).Location and timestamp — 
“This fault will not clear permanently until the root cause is resolved. Driving with this issue for extended periods may damage the catalytic converter and increase operating costs.”Persistence note — 
This means the enrichment data already exists somewhere in the backend — the front-end is rendering it. The API work is therefore not about building a DTC lookup table from scratch; it’s about exposing the same data source the front-end already uses through the external API.
Current API Response (from plain-English reference)

Implementation Approach
Since the front-end already renders this data, the implementation path is:
Identify the data source the front-end uses for DTC enrichment. This could be a database table, an internal service, or a static lookup. The front-end notification panel for P1094 shows content that’s too specific and well-structured to be generated on-the-fly — it’s being looked up from somewhere.
Wire the same data source into the external API’s DTC endpoint. The controller should call the same service/lookup the front-end uses, ensuring parity between what partners see via the API and what fleet operators see in the dashboard.
Flatten the response structure from the current array-of-arrays to a single array of enriched code objects (as shown above).
Add the location data from the position record at the time the fault was detected. The front-end already shows this (“11 Gould Rd, Herston QLD AUS”) so the data is available.
Handle unknown codes gracefully — for manufacturer-specific codes not in the lookup, return the raw code with null enrichment fields. Don’t omit the code entirely.

3. VIN Lookup Capability
Open questions: New endpoint? Should lookups be scoped to the company’s own vehicles?
Answer: Strongly recommend scoping to company vehicles only. Allowing cross-company lookups creates a data leakage risk where one customer could enumerate another’s fleet. This is a security concern and a potential privacy issue under Australian Privacy Principles.
Endpoint Design
Combine VIN lookup with the vehicle details endpoint (see Feature 5). A single vehicle metadata endpoint that accepts multiple lookup keys:
POST /api/GetVehicleDetails
Query params (at least one required):
  id    = 85          -- vehicle ID
  vin   = 1HGCM82633A004352  -- VIN

1. Rate Limiting with Clear Headers
The current API has no visible rate limiting or feedback to developers about their usage. Adding standard rate limit headers lets integrators build resilient applications without guessing.

🤖 Claude Code: Add Rate Limit Headers
Add ASP.NET Core rate limiting middleware using the built-in RateLimiter. Configure a sliding window policy per API key. Add X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset headers to every response. Return 429 with Retry-After when exceeded.
2. Auto-Generated SDK / Client Library
Developers currently hand-roll HTTP calls against the API. An auto-generated client library eliminates boilerplate and reduces integration time from days to hours.
Use NSwag (for C#/.NET) or openapi-generator (for JavaScript, Python, Java) to auto-generate typed client libraries directly from openapi.yaml.Generate from the OpenAPI spec —

🤖 Claude Code: Generate SDK
Read our openapi.yaml and generate a TypeScript client using openapi-generator-cli. Output to /sdk/typescript/. Include typed request/response interfaces, error handling, and automatic token refresh.
3. Idempotency Keys for Commands
Commands like Engine Disable and Enable have real-world consequences. Network failures can leave developers unsure if a command was received. Idempotency keys solve this.
On all command endpoints (SendIgnitionCommand, SetAlertAsRead). If the same key is sent twice within a 24-hour window, return the original response without executing the command again.Accept an Idempotency-Key header — 
A simple key-value store (Redis or database table) mapping idempotency keys to response payloads. Auto-expire after 24 hours.Store keys server-side — 
A developer retrying a failed engine disable request must not accidentally send the command twice. This is a safety feature as much as a convenience one.Critical for engine commands — 
🤖 Claude Code: Implement Idempotency
Create an IdempotencyMiddleware that intercepts POST requests with an Idempotency-Key header. Store the key and response in Redis with a 24h TTL. If the same key is seen again, return the cached response with a 200 and an X-Idempotent-Replay: true header.

8. Cursor-Based Pagination
The Phase 1 sprint adds offset-based pagination (page/pageSize), which is a good start. For real-time telemetry data where records are constantly being added, cursor-based pagination is more reliable.
Base64-encode the last record’s sort key (e.g. timestamp + ID). Clients pass this as a ?cursor parameter to get the next page.Use an opaque cursor instead of page numbers — 
With offset pagination, new records arriving between page requests can shift the result set. Cursors are immune to this because they anchor to a specific position in the data.Prevents skipping or duplicating records —
